# ADR-0001: 采用分层架构

- **状态**：Accepted
- **日期**：2026-01-01
- **决策者**：Mady Authors
- **影响范围**：全体代码架构

## 上下文

Mady 是一个面向专利/法律专业人士的智能 Agent 平台。系统需要同时支持多种外部协议（A2A、MCP、HTTP/SSE）、多种 LLM 提供者（DeepSeek、智谱、Kimi 等）、多种 UI 形态（终端 TUI、Web UI），还需要容纳领域特定逻辑（专利分析、法律比较）和高级特性（心理引擎、图执行）。

在设计之初，面临着组织这样一个复杂系统的挑战。如果没有清晰的架构分层，代码将迅速变得难以维护。

## 决策

采用如下 6 层分层架构，严格的依赖方向是「上层依赖下层，下层不依赖上层」：

```
外部接口层：  A2A | A2UI | Server | AGUI | MCP | ACP
                  \       |       |       |     /
                   核心引擎层：agentcore
                   /    |    \      \
               提供者层   工具层   扩展层
                   \       |
                    基础设施层：graph/ session/ skill/ prompt/ store/
                                graph/ disclosure/ memory/ filequeue/ fuzzy/
                                   |
                   领域层：domains/ (chat | patent | legal)
                                   |
                   心理层：psychological/
```

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **分层架构（已选）** | 依赖方向清晰，模块边界明确，易于测试和替换 | 层间通信有开销 |
| Hexagonal (Ports & Adapters) | 极佳的隔离性和可测试性 | 过多的抽象层，不适合此项目的规模 |
| 扁平包结构 | 简单直观 | 随规模增长迅速混乱 |

## 后果

### 正面影响
- agentcore 是纯 Go 代码，仅有 `gorilla/websocket` 一个外部依赖
- 协议层（A2A、MCP）可独立演进
- TUI 内部有 8 个子层（Layer 0-7），严格遵守依赖方向，实现了完全解耦
- 工具层是独立 Go Module，可供外部项目直接使用

### 负面影响
- 新开发者需要时间理解分层逻辑
- 跨层变更有时需要修改多个包
- 接口适配器带来一定代码量

### 迁移策略
无需迁移——这是项目从零开始就采用的架构。
