# P3 专家盲测方案（数据收集就绪版）

> 状态：方案成文备用，**当前不启动真实盲测**（2026-07-17 路线图共识：P3 缓行，只做数据收集就绪）。
> 前置依赖：HITL 触点留痕链路（已就绪，见 §7）、P2A 全量 31 题 live 基线（触发方式见 §8）。
> 方法论依据：`docs/evaluation-baseline-v0.7.md` §2c —— LLM 模拟 HITL 存在方法论困境（mockHumanRevision 对正确初稿也会改坏，L5 得分 0.320 为五层最低），**HITL 价值必须真人真测**，无法用 LLM 代理估计。

---

## 1. 盲测目标

测量 Mady Agent 产出在真实专业人员工作流中的**人工采纳率 / 修改率 / 拒绝率**，并估算人机协作的生产力增益。

**对齐北极星指标**：

> 每个专业工作成果中，经人工确认、可定位到权威来源且无需结构性重做的证据化结论数量 ÷ 专业人员投入时间。（`docs/roadmap.md` §一）

盲测要回答的三个问题：

1. **质量**：Agent 的结论单元有多大比例被专家原样采纳（adopted）、修改后采纳（modified）、拒绝（rejected）？
2. **效率**：专家在 Agent 辅助下，单位时间产出的"经确认证据化结论数"是否显著高于纯人工作业？
3. **信任校准**：专家的采纳决策是否与结论的实际正确性一致（防止"全盘照收"或"全盘不信"两种失败模式）？

## 2. 核心度量定义

以**结论单元**（见 §5.2）为计数粒度：

| 指标 | 定义 |
|------|------|
| 采纳率 | adopted 结论单元数 ÷ 全部已标注结论单元数 |
| 修改率 | modified 结论单元数 ÷ 全部已标注结论单元数 |
| 拒绝率 | rejected 结论单元数 ÷ 全部已标注结论单元数 |
| 有效采纳率 | (adopted + modified) ÷ 全部已标注结论单元数 |
| 权威来源可定位率 | 被专家标记"可定位权威来源"的 adopted/modified 单元 ÷ adopted+modified 单元数 |
| 人效（北极星分子） | 经确认且可定位来源的结论单元数 ÷ 专家投入小时数 |
| 人效增益 | （Agent 辅助人效 − 独立作业人效）÷ 独立作业人效 |
| 标注一致性 | 双人标注案件的 Cohen's κ（采纳/修改/拒绝三分类） |

## 3. 案件来源（10 个案件）

**红线**（`SECURITY.md` / `AGENTS.md`）：禁止在测试数据中使用真实案件文件、真实当事人信息。

按优先级排序的来源建议：

1. **已公开的无效宣告请求审查决定书（首选，5-6 件）**
   国家知识产权局复审无效部官网公开的决定书天然完成脱敏（当事人信息已依法公开披露，不属于"内部真实案件文件"）。参照 P2B 重建的成功经验（从 31562 件原始决定书提取案例，见 `docs/roadmap.md` §P2B 更正条目）。选择标准：
   - 结论分布配额：全部无效 / 部分无效 / 维持有效 均衡取样，避免退化分布；
   - 难度分层：简单（单一新颖性理由）2 件、中等（创造性 + 证据组合）2-3 件、复杂（多理由多证据）1 件；
   - 技术领域分散（机械 / 电学 / 化学至少各 1 件）。
2. **虚构技术交底书（3-4 件）**
   由 LLM 辅助起草 + 非参评代理人人工改写定稿，覆盖交底书 → 新颖性初评场景（对应 disclosure 管线的首要用户旅程）。虚构件必须：
   - 不映射任何真实申请人 / 发明人 / 申请号；
   - 技术方案为公知技术的显而易见组合或经典教科书案例改编，确保"参考答案"无争议；
   - 入库前由 2 名非参评代理人做事实核查。
3. **内部真实案件（默认不用）**
   仅当同时满足：客户书面授权 + 完全脱敏（去除当事人、申请号、代理人、日期锚点）+ 法务复核通过，才可作为补充。本方案默认不启用。

案件包交付物：每案一个独立目录，含案件材料（决定书/交底书）、给专家的《任务书》（要回答的问题清单）、给分析员的《参考答案与评分要点》（专家不可见）。

## 4. 专家画像与人数

| 项目 | 要求 |
|------|------|
| 资格 | 执业专利代理人，≥ 5 年实务经验 |
| 经验 | 独立承办无效宣告 / 撰写答辩案件 ≥ 20 件 |
| 人数 | 目标 4-6 人；最低可行 3 人（低于 3 人统计功效不足，结论只能作定性参考） |
| 多样性 | 至少覆盖 2 个技术领域背景；不得全部来自同一代理机构 |
| 回避 | 与任一案件存在利害关系者回避该案件；参与过案件包制作者不得参评 |

报酬建议：按件计酬，单案预算对应其正常计费时长的 2-2.5 倍（含独立作答 + 标注两阶段），并在知情同意书中写明。

## 5. 盲测流程（对照设计）

### 5.1 总体设计：先后对照 + 盲化标注

每位专家对每案经历两个阶段（先独立、后标注），从机制上消除"看了 AI 答案后反过来影响自己基线"的回溯污染：

```
T0 培训校准（30 min，统一标注规则与工具使用）
 │
 ├─ T1 独立作答（盲基线）：专家在看不到任何 Agent 产出的前提下，
 │    按《任务书》完成分析，提交自己的结论清单，记录起止时间。
 │
 ├─ （强制休息 ≥ 15 min，重置工作记忆）
 │
 └─ T2 盲化标注：专家看到该案的"产出报告"，对其中逐条结论单元做
      采纳 / 修改 / 拒绝判定，填写修改文本与理由，记录起止时间。
```

**盲的实现**：T2 阶段专家不被告知产出来源构成。10 个案件中混入 1-2 个**对照案件**——其"产出报告"实为另一位非参评资深代理人的匿名作答（sham 对照）。若专家对"AI 标签"存在系统性偏见（对 AI 产出更苛刻或更宽容），sham 案件的标注分布会与真 Agent 案件出现可检测差异；无显著差异则说明标注反映的是内容质量而非来源偏见。

### 5.2 结论单元（标注粒度）

把每份产出报告预先拆解为**结论单元**：一条可独立判定的最小结论，包含

- 结论陈述（如"权利要求 1 相对于证据 1 不具备新颖性"）；
- 依据法条（如 A22.2）；
- 引用证据及其定位（证据编号 + 段落/附图）。

拆解工作由分析员在盲测前完成并编号（CU-01、CU-02…），专家对单元整体判定。拆解粒度对齐 P2B 评估中 llm_judge 的 rubric 维度（conclusion / reasoning / citation），保证盲测数据可与自动评估指标交叉验证。

### 5.3 案件分配与顺序控制

- 每案由 **≥ 2 名专家**独立走完整流程（用于 κ 一致性）；
- 专家 × 案件分配采用拉丁方设计，平衡案件顺序与学习效应；
- 每人每半天不超过 2 案，避免疲劳污染后段数据。

### 5.4 流程时间线

| 阶段 | 内容 | 产出 |
|------|------|------|
| 准备（1 周） | 案件包制作、评价表预填表头、环境就绪（approvals.db 清空归档） | 案件包 ×10、评价表 |
| 先导（pilot，2 天） | 1 名专家 × 2 案走完全流程，校准任务书与标注规则 | 修订后的任务书/规则 |
| 正式（1-2 周） | 全量 10 案 × 4-6 专家 | 原始标注数据 |
| 分析（3 天） | 统计 §2 全部指标、κ、sham 对照检验 | 盲测报告 |

## 6. 数据采集字段（对齐 RecordDecision schema）

盲测数据分两层落库，用 `case_id` + 结论单元编号关联：

### 6.1 触点决策层（自动，approvals.db）

专家对 Agent 产出在系统内做采纳/修改/拒绝操作时，走现有 `domains.ApprovalRecord` 留痕链路，字段对齐：

| ApprovalRecord 字段 | 盲测语义 |
|---------------------|----------|
| `ID` | 记录唯一 ID（自动生成） |
| `SessionID` | 盲测场次 ID（如 `blind-e2-case05`） |
| `CaseID` | 案件编号（如 `case-05`） |
| `Timestamp` | 决策时间（用于耗时分析） |
| `TriggerKeyword` | 触点来源：`review`（TUI 软中断）/ `disclosure_review`（交底书复核）/ `tool_permission:*`（ACP 授权） |
| `OriginalOutput` | 被审产出原文（TUI 硬中断路径含 gate 标识与报告摘要） |
| `Decision` | adopted / modified / rejected |
| `ModifiedOutput` | 专家修改后的文本（Decision=modified 时必填） |
| `Feedback` | 修改/拒绝理由 |
| `State` | 审批状态机状态（approved/modified/rejected，由 Decision 推导） |

### 6.2 结论单元层（人工填表，评价表）

结论单元粒度的标注用评价表采集（当前 RecordDecision schema 不含结论单元 ID 与耗时字段，**不为此扩 schema**——单元 ID 写入 `Feedback` 前缀约定 `[CU-03] 理由…` 完成关联，实施阶段若证实需要再评估 schema 演进）。

### 6.3 评价表模板（每案一表）

| 字段 | 填写人 | 说明 |
|------|--------|------|
| 专家代号 | 分析员预填 | E1-E6 匿名代号，不出现真实姓名 |
| 案件编号 | 分析员预填 | case-01 … case-10 |
| 是否 sham 对照 | 分析员预填（对专家保密） | 是 / 否 |
| T1 起止时间 | 专家 | 独立作答耗时 |
| T1 独立结论清单 | 专家 | 专家自己的结论（编号与 CU 对应） |
| CU 编号 | 分析员预填 | 结论单元 ID |
| CU 判定 | 专家 | 采纳 / 修改 / 拒绝 |
| 修改后文本 | 专家 | 判定为"修改"时必填 |
| 判定理由 | 专家 | 判定为"修改/拒绝"时必填，按以下分类勾选 + 补充：法条适用错误 / 事实遗漏 / 推理缺陷 / 证据引用错误 / 表达不清 / 其他 |
| 权威来源可定位 | 专家 | 是 / 否（该结论是否能定位到具体法条 + 证据段落） |
| T2 起止时间 | 专家 | 标注耗时 |
| T3 问卷 | 专家 | 信任度量表（1-5）、SUS 可用性量表、开放式反馈 |

## 8. 质量锚点：P2A 全量 31 题 live 基线

盲测启动前必须以 P2A 31 题全量跑批锚定当时的模型质量（分数大幅波动时需先归因再启动盲测）：

```bash
MADY_LIVE_EVAL=1 MADY_EVAL_SUITE=p2a DEEPSEEK_API_KEY=<key> \
  go test -run 'TestLiveAgentBaselineEval' -v ./evaluate/benchmark/
```

（`MADY_EVAL_SUITE=p2a` 为全量 31 题稳定顺序筛选；`MADY_EVAL_CASES=31` 亦可，但为随机顺序。无 DeepSeek key 时用 `MADY_EVAL_API_KEY` + `MADY_EVAL_BASE_URL` + `MADY_EVAL_MODEL` 指向任意 OpenAI 兼容端点。）

参考锚点：P2A L1（Agent 框架）历史基线 llm_judge 均值 0.700（10 题抽样，DeepSeek，`docs/evaluation-baseline-v0.7.md`）。

**全量 31 题基线（2026-07-17 首次固化，`docs/evaluation-baseline-v0.8.md`）**：本地 oMLX `gemma-4-12B-it-8bit`（推理与评判同模型自评，MaxTokens 8192）——L0 裸 LLM 通过率 90.3%（28/31，citation 0.935 / judge 0.723）；L1 Agent 框架通过率 93.5%（29/31，citation 0.935 / judge 0.746）；L3 +检索工具通过率 90.3%（28/31，citation 0.935 / judge 0.798，**工具全程零触发**）。共性失败题 2 道（`2008_a31_02`、`2009_a22_01`），归因为小模型法条编号幻觉与基准口径疑点；L3 未修复幻觉（Agent 不主动检索）。注意：与 DeepSeek 历史基线不可直接对比（被测模型与 judge 均不同），盲测启动时须以同口径复跑锚定。

## 9. 成功标准（通过线）

以下为**预设判定线**，pilot 后允许一次性校准并记录校准理由：

| 维度 | 通过 | 观察（需归因） | 不通过 |
|------|------|----------------|--------|
| 采纳率（adopted） | ≥ 60% | 40-60% | < 40% |
| 有效采纳率（adopted+modified） | ≥ 85% | 70-85% | < 70% |
| 权威来源可定位率 | ≥ 90% | 75-90% | < 75% |
| 人效增益 | ≥ +30% | +10% ~ +30% | < +10% |
| sham 对照偏差 | 无显著差异（χ² p > 0.05） | — | 显著来源偏见，结论降级为定性 |
| 标注一致性 κ | ≥ 0.6 | 0.4-0.6（加评第三人） | < 0.4（标注规则返工） |

补充判定原则：

- **拒绝内容的构成**比拒绝率绝对值更重要：若拒绝集中在"证据引用错误"，属检索层问题；集中在"法条适用"，属推理层问题——分别对应不同的修复路线。
- 出现 ≥ 1 例"专家采纳了实际错误的结论"（采纳后分析员复核发现硬伤）即触发信任校准专项分析，即使整体采纳率达标。
- 采纳率 > 95% 且修改率 ≈ 0 时警惕"全盘照收"：需结合 T3 信任度量表与 sham 案件表现判断是否为过度信任。

## 10. 伦理与免责

1. **知情同意**：参评前签署知情同意书，明确告知：数据仅用于产品评估研究、标注过程被记录、可随时无理由退出且不影响已发生报酬。
2. **来源盲化与揭盲**：sham 对照的存在在知情同意书中概括披露（"部分材料可能来自不同来源"），但不告知具体是哪几案；研究结束后向专家揭盲。
3. **匿名化**：一切产出物只使用专家代号（E1-E6）；报酬发放记录与研究数据物理分离存放。
4. **AI 产出免责**：案件包与评价表显著位置标注"AI 产出仅供评估研究，不构成法律意见，不得用于任何在办案件"。对齐 `docs/tone-style-guide.md` 的免责声明约定。
5. **数据合规**：案件材料仅使用 §3 允许的公开/虚构来源；研究数据保存期限 2 年，到期删除可关联原始材料。
6. **无绩效用途**：标注结果不用于评价专家个人业务能力，不与报酬挂钩（报酬只与完成度挂钩），避免激励扭曲标注行为。

## 11. 风险与对策

| 风险 | 对策 |
|------|------|
| 锚定效应（先看 AI 答案影响独立基线） | T1 先独立作答且提交后不可改；T1/T2 间强制休息 |
| 学习效应（后段案件越做越快） | 拉丁方案件顺序；分析时纳入顺序协变量 |
| 案件难度不均导致指标失真 | 难度分层配额；按层分别报告指标 |
| 专家招募不足（< 3 人） | 降级为定性案例研究，不报告统计结论 |
| 模型版本漂移 | 盲测期间锁定模型与 prompt 版本；P2A 31 题基线前后各跑一次 |
| 评价表与 approvals.db 关联失败 | CU 编号写入 Feedback 前缀约定；每日核对当日数据 |

## 12. 与既有评估体系的关系

- **P2A / P2B（自动评估）**：度量"答案对不对"（对齐参考答案），可高频跑批；盲测度量"专家认不认、快不快"，是 L5 HITL 价值的唯一有效测量手段（`docs/evaluation-baseline-v0.7.md` §2c 结论）。
- **AdoptionRate 指标（A3）**：盲测期间 approvals.db 累积的真实决策记录即 AdoptionRate 的生产数据源；盲测结束后，评价表数据可回放为 Golden Benchmark 的第二层转换输入（C1 管线）。
- **北极星指标**：盲测的人效与可定位率是北极星的首次直接测量，结果应回填 `docs/roadmap.md` 的 P3 条目。
