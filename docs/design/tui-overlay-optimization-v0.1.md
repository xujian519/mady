# Mady TUI + 浮层优化方案 v0.1

> 状态：设计待评审 | 日期：2026-07-20
>
> 适用范围：当前单主视图 `TUI`，以及现有 `Overlay` 浮层体系的体验优化
>
> 相关现状参考：
> - 已归档的大改版方案：`docs/archive/tui-design-plan.md`
> - 当前设计索引：`docs/design/README.md`

## 1. 背景

Mady 当前 `TUI` 已具备可用的单主视图聊天壳、会话管理、状态栏和浮层机制，能够支撑日常交互与专业工作流推进。但随着 `session`、`approval`、`disclosure`、`todo`、`settings` 等能力逐步补强，界面正在面临两个相反的风险：

- 一类风险是信息过密：把更多状态、日志、证据、待办直接堆回主界面，导致用户难以把握当前主任务。
- 另一类风险是结构失衡：为了容纳更多能力，过早引入两栏/三栏工作台，导致焦点管理、键位语义和状态同步复杂化。

本方案明确选择一条更克制的路线：**保持单主视图结构不变，优先优化主界面的判断主轴，并把复杂性收束到按需开启的浮层中。**

## 2. 目标与非目标

### 2.1 目标

1. 把当前主界面从“聊天承载页”提升为“当前判断页”
2. 把现有浮层从“临时弹窗集合”提升为“第二层认知空间”
3. 统一 `TUI` 的状态语言、键位语义和浮层分类，降低认知切换成本
4. 让“复核门”成为关键动作前的核心体验，体现“先明其据，再断其行”
5. 在不引入两栏/三栏布局的前提下，显著改善专业工作流的可见性与可复核性

### 2.2 非目标

- 不启动新的 Sidebar / 多栏工作台重构
- 不引入可拖拽 pane、底部常驻日志瀑布、复杂分屏布局
- 不在本阶段重做主题系统或做大规模视觉风格切换
- 不修改后端任务、审批、会话存储语义；本方案聚焦现有 `TUI` 壳层呈现方式

## 3. 设计总原则

### 3.1 主轴唯一

一屏只突出一个主任务，不并列多个同权重点。主界面优先回答：

1. 当前处于什么阶段
2. 当前判断是什么
3. 当前仍缺什么
4. 下一步最合理的动作是什么

### 3.2 默认保守

默认仅显示推进当前任务所必需的信息。证据细节、系统日志、冲突说明、引用上下文等信息不默认铺开，但必须一键可达。

### 3.3 解释后置但不可缺席

主界面不承担全部解释责任，但每条关键判断都应可追溯到依据、风险和系统条件。浮层承担“解释空间”的角色，而不是简单的确认弹窗。

### 3.4 复核先于执行

高价值动作前不直接强化提交按钮，而是先引导用户完成一次最小必要复核。界面要帮助用户“再看一眼”，而不是推动用户“马上点下去”。

### 3.5 系统透明但不扰民

降级、阻塞、失败、审批挂起等状态必须真实可见，但不应持续侵占主界面注意力。系统态优先采用摘要化呈现，细节进入专门浮层。

## 4. 现有结构判断

当前 `TUI` 的核心优势是：

- 单主视图结构稳定，认知入口清楚
- `Overlay` 已经是成熟机制，适合承接第二层信息
- 会话、审批、任务、系统状态已有后端基础，不需要先扩 UI 架构才能使用

当前主要问题是：

- 主界面仍偏“聊天流承载”，缺少“当前判断主轴”
- 浮层缺少清晰分工，不同类型的内容容易混成“都可以弹出来”
- 关键动作前的复核体验不足，容易退化为普通确认
- 状态词、动作词、键位语义还未形成统一秩序

因此，本方案不改变整体信息结构范式，而是对现有单主视图做一次“收束式优化”。

## 5. 主界面优化方案

## 5.1 主界面职责

主界面只承担四项职责：

1. 显示全局状态
2. 展示当前判断
3. 摘要当前上下文
4. 提供下一步动作入口

任何不直接服务以上四项职责的信息，都应优先移入浮层。

## 5.2 推荐结构

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Mady · 项目: XXX · 阶段: awaiting_review · 模式: degraded           │
├──────────────────────────────────────────────────────────────────────┤
│ 当前判断                                                            │
│ 某技术点可形成初步结论，但证据完整性仍不足。                        │
│                                                                      │
│ 当前置信度: 中                                                      │
│ 仍待确认: 引用来源 1 条、对比材料 2 份                              │
├──────────────────────────────────────────────────────────────────────┤
│ 当前上下文                                                          │
│ - 已归拢证据 6 条                                                    │
│ - 其中核心证据 2 条                                                  │
│ - 存在冲突证据 1 条                                                  │
│ - 最近一次系统事件: Provider 已降级为 fallback                       │
├──────────────────────────────────────────────────────────────────────┤
│ [r] 进入复核   [e] 查看证据   [s] 系统态   [c] 切换会话   [/] 命令   │
└──────────────────────────────────────────────────────────────────────┘
```

## 5.3 主界面字段约束

### 当前判断

- 必须短，可一眼读完
- 必须对应当前阶段，而不是泛泛总结整段历史
- 若暂不能得出结论，必须明确写出“当前尚不足以下判断”

### 当前置信度

- 仅使用少量离散等级，如 `低 / 中 / 高` 或 `初步 / 稳定 / 待核验`
- 不使用伪精确数字化表达

### 仍待确认

- 仅保留 1 至 3 条最影响推进的未决项
- 优先展示阻碍动作执行的缺口

### 当前上下文

- 只保留摘要，不展开长日志、长推理链、长原文
- 若存在系统事件，仅展示对当前判断真正有影响的一条摘要

## 5.4 主界面禁止项

- 不长期显示完整日志流
- 不默认铺开大段证据原文
- 不让聊天历史吞没“当前判断”区域
- 不把所有次级动作都放到底部动作带
- 不把所有系统异常都提升为强警报

## 6. 浮层分类

本方案将浮层分为四类，并要求每类浮层承担清晰职责。

### 6.1 选择型浮层

用途：切换对象，而不是审阅内容。

典型场景：

- 会话切换
- 线程切换
- 分支切换
- 工作对象选择

设计要求：

- 打开快，退出快
- 支持过滤与快速定位
- 选中后立即返回主界面
- 不承载长文本与高风险确认

### 6.2 审阅型浮层

用途：查看依据与细节。

典型场景：

- 证据详情
- 引用上下文
- 冲突证据对照
- 推理链展开

设计要求：

- 以当前焦点对象为中心
- 支持对象间切换
- 支持回到主界面继续推进
- 不承担主执行入口

### 6.3 复核型浮层

用途：关键动作前完成一次结构化审阅。

典型场景：

- 人工复核
- 输出前检查
- 高风险确认

设计要求：

- 必须清单化
- 必须显示不确定性与风险
- 必须提供“返回补材料”的明确出口
- 不能退化成只有“确定/取消”的普通确认框

### 6.4 系统型浮层

用途：解释系统运行条件，而不是承载业务判断。

典型场景：

- 日志摘要
- 降级说明
- 阻塞原因
- 当前配置快照

设计要求：

- 默认摘要化，细节可展开
- 明确展示“这件事对当前任务的影响”
- 不长期常驻主界面

## 7. 核心浮层规范

## 7.1 证据详情

目标：让用户知道“这条证据是什么、为何重要、是否可靠、是否足以支撑当前判断”。

推荐结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 证据详情 · A-03                                              │
├──────────────────────────────────────────────────────────────┤
│ 标题: 文献 A 第 3 节                                         │
│ 来源: 已定位原文                                             │
│ 角色: 核心证据                                               │
│                                                              │
│ 摘要                                                         │
│ 这条证据用于支持“某技术点存在先例”的判断。                   │
│                                                              │
│ 原文片段                                                     │
│ "... ..."                                                    │
│                                                              │
│ 使用说明                                                     │
│ - 支撑判断: 初步结论 1                                       │
│ - 当前风险: 上下文仍待补足                                   │
│ - 建议动作: 对照冲突证据后再进入复核                         │
├──────────────────────────────────────────────────────────────┤
│ [j/k] 切换证据   [c] 比较   [m] 标记待确认   [Esc] 返回      │
└──────────────────────────────────────────────────────────────┘
```

必备字段：

- 标题
- 来源状态
- 证据角色
- 摘要
- 原文片段
- 使用说明

其中“使用说明”必须明确这条证据对当前判断的作用，而不是只展示原文。

## 7.2 复核门

目标：在关键动作前完成一次“依据、风险、下一步”的最小必要审查。

推荐结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 复核门                                                       │
├──────────────────────────────────────────────────────────────┤
│ 当前判断                                                     │
│ 某技术点可形成初步结论，但不宜直接对外输出。                 │
│                                                              │
│ 主要依据                                                     │
│ - 核心证据 A                                                 │
│ - 对照材料 B                                                 │
│ - 引用记录 C                                                 │
│                                                              │
│ 复核清单                                                     │
│ [x] 已列明主要依据                                           │
│ [x] 已标注不确定性                                           │
│ [ ] 冲突证据已处理                                           │
│ [ ] 风险表述已收敛                                           │
│ [x] 下一步动作清楚                                           │
│                                                              │
│ 风险提示                                                     │
│ - 当前有 1 条引用待人工核验                                  │
│ - 当前结论适合内部研判，不宜直接外发                         │
├──────────────────────────────────────────────────────────────┤
│ [p] 通过复核   [b] 返回补证据   [f] 标记阻塞   [Esc] 返回    │
└──────────────────────────────────────────────────────────────┘
```

设计要求：

- `通过复核` 不能成为唯一强势出口
- `返回补证据` 必须是同等清晰的路径
- 风险提示必须直陈，不以措辞弱化不确定性
- 复核结果应与当前阶段和后端审批/评审状态保持一致

## 7.3 系统态

目标：让运行条件透明，但不打断主流程。

推荐结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 系统态                                                       │
├──────────────────────────────────────────────────────────────┤
│ 当前运行                                                     │
│ - 模式: degraded                                             │
│ - 原因: Provider 不支持 json_schema                          │
│ - 策略: 已切换 fallback                                      │
│                                                              │
│ 最近事件                                                     │
│ - 18:41 MCP 发现超时，已跳过                                 │
│ - 18:42 判断生成成功，进入 awaiting_review                   │
│ - 18:43 引用校验发现 1 条待确认                              │
│                                                              │
│ 当前影响                                                     │
│ - 输出仍可继续                                               │
│ - 结构化格式能力受限                                         │
├──────────────────────────────────────────────────────────────┤
│ [l] 详细日志   [Esc] 返回                                    │
└──────────────────────────────────────────────────────────────┘
```

设计要求：

- 明确区分“事实状态”和“对当前任务的影响”
- 默认仅显示最近且相关的事件
- 长日志进入下一层，不直接占满当前浮层

## 8. 键位规范

为避免不同浮层产生混乱语义，本方案统一以下高频键位。

| 键位 | 语义 | 适用范围 |
|------|------|----------|
| `Tab / Shift-Tab` | 在主界面交互区之间切换 | 主界面 |
| `Enter` | 打开当前焦点项或确认低风险动作 | 主界面 / 浮层 |
| `Esc` | 关闭当前浮层或返回上一级 | 全局 |
| `/` | 打开命令或搜索入口 | 全局 |
| `j / k` | 上下移动焦点项 | 列表 / 审阅型浮层 |
| `r` | 进入复核 | 主界面 |
| `e` | 查看证据 | 主界面 |
| `s` | 打开系统态 | 主界面 |
| `c` | 切换会话或比较当前项 | 主界面 / 审阅型浮层 |
| `l` | 查看日志摘要或详细日志 | 系统型浮层 |

约束：

- 单键只承载高频、低歧义动作
- 高风险动作不得直接一键提交，应先进入复核型浮层
- 主界面负责“推进”，浮层负责“审阅与解释”

## 9. 状态语言统一

全 `TUI` 建议统一采用以下状态集：

- `idle`
- `running`
- `awaiting_input`
- `awaiting_review`
- `ready`
- `blocked`
- `degraded`
- `done`
- `failed`

显示原则：

- 状态名后必须跟一行简明解释
- 必须说明当前是否可推进，以及推进受何影响

示例：

- `awaiting_review`：结论已生成，尚待人工确认
- `degraded`：运行可继续，但部分能力已降级
- `blocked`：当前缺少必要条件，暂不能推进

## 10. 文案规范

推荐句式：

- `当前判断：`
- `已有依据：`
- `仍待确认：`
- `建议下一步：`
- `当前影响：`
- `降级说明：`

避免句式：

- “我已经帮你想好了”
- “最佳答案如下”
- “完美完成”
- “强烈建议立即执行”

要求：

- 用语偏工作台，不偏陪伴聊天
- 对不确定性直说，不用情绪化表述掩盖风险
- 成功提示强调结果，不渲染情绪

## 11. 实施边界

为保持改动可控，本方案建议拆成 4 个小阶段，每阶段均可独立评审与回滚。

### 阶段一：收紧主界面

- 强化“当前判断”和“下一步动作”可见性
- 减少聊天历史对主判断区的挤压
- 将非主轴信息迁出主界面

### 阶段二：统一浮层分类

- 为选择型、审阅型、复核型、系统型浮层建立统一标题、尺寸、关闭规则
- 清理“同类内容不同浮层风格”的问题

### 阶段三：做强复核门

- 将复核从“确认动作”提升为“结构化审阅动作”
- 明确与审批/评审后端状态的衔接关系

### 阶段四：补系统态浮层

- 将降级、阻塞、日志摘要集中展示
- 保持主界面只展示必要的系统影响摘要

## 12. 验收标准

本方案完成后，界面应满足以下标准：

1. 用户进入主界面后，能在 3 秒内看懂当前状态、当前判断和下一步动作
2. 用户查看证据时，能同时知道证据内容、证据作用和证据风险
3. 用户执行关键动作前，必经一次清晰复核，而不是直接提交
4. 用户能够看到降级、阻塞、异常对当前任务的真实影响
5. 整体使用感受应为“稳、明、可复核、不过载”

## 13. 与已归档大改方案的关系

`docs/archive/tui-design-plan.md` 记录了更偏“导航化、多栏化”的长期演进思路；本方案不是其继续执行，而是一个更贴近当前架构与项目阶段的收束版替代路径。

两者关系如下：

- 归档方案解决的是“如何把 TUI 演进为更完整工作台”
- 本方案解决的是“在现有单主视图结构下，怎样把体验先做稳、做清楚”

因此，本方案优先级更高，也更适合作为近期落地依据。
