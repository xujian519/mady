## 变更概要

v0.7 → v0.8 的核心转变：**从「10 题抽样」扩到「P2A 全量 31 题」，并从付费 DeepSeek 切换到本地 MLX 免费端点**。

- 首次完成 P2A 全量 31 题（真题）两层级（L0 裸 LLM / L1 Agent 框架）跑批，回应 v0.7「小样本陷阱」遗留问题（3→10 题结论多次反转，需全量稳健基线）
- 推理与评判均改用本地 oMLX 端点（`http://127.0.0.1:8000/v1`，模型 `gemma-4-12B-it-8bit`），**零 API 成本**，可任意高频复跑
- 跑批 harness 工程增强（见末节）：生成/评判双阶段并发、按题落盘缓存（断点续跑）、单题 15 分钟超时、MaxTokens 8192 上限、修复缓存读写数据竞争

### 与历史基线的不可比性声明

**本报告数据与 v0.5/v0.7 的 DeepSeek 基线（如 v0.7 L1 10 题 llm_judge 均值 0.700）不可直接横向对比**，原因：

1. **被测模型不同**：v0.7 测的是 DeepSeek `deepseek-chat`（云端大模型）裸能力；v0.8 测的是本地 12B 小模型。两者分差反映模型差距，而非框架变化。
2. **Judge 不同**：v0.7 由 DeepSeek 评判，v0.8 由 gemma-4-12B **自评**。小模型自评存在宽松偏差（其自身无法校验法条编号正确性，见失败题分析），分数系统性偏高。
3. **长度上限**：8192 tokens 截断在 DeepSeek 基线中不存在。

v0.8 基线的用途是：**①为本地免费高频回归提供锚点（每次改动后复跑同口径对比自身）；②为 P3 盲测提供启动时的质量锚定（`p3-blind-test-plan.md` §8）；③暴露小模型短板作为优化靶点。** 需要与云端大模型对比时，应以相同 judge 重跑而非直接比较本表数字。

## 关键发现

**1. L1 ≥ L0 在全量 31 题上成立，Agent 框架无退化且略有增益**

全量口径下 L1 通过率 93.5% ≥ L0 90.3%，judge 均值 +0.023；31 题中 13 题 L1 更高、12 题持平、6 题微降（均在 ±0.08 内的边际波动）。v0.7 在 10 题抽样上的「L0→L1 稳定增益」结论在全量集上得到确认，且增益幅度温和。

**2. 两道共性失败题（两层均 FAIL）归因：小模型法条编号幻觉**

- `patent_exam_2008_a31_02`（分案申请专题，citation 两层均 0）：期望引用「第33条 / 实施细则第42条」，而模型实际输出引用**「专利法第四十七条（分案申请）」「实施细则第二十条/二十一条」——法条编号系幻觉**（分案申请的依据是实施细则第42条；专利法第47条是无效效力条款）。值得注意的是 llm_judge 给了 0.77–0.87 的不错分数：**同模型自评无法识别法条编号错误**，分析框架「说得像」即得高分。citation 规则抽取指标成为唯一防线。
- `patent_exam_2009_a22_01`（无效实务，citation 0.333）：期望引用「第31条 / 实施细则第64条 / 第65条」。该题存在基准口径疑点——参考答案要点自身引用的是「实施细则第68条 + 审查指南第四部分第三章」，与期望引用集合不完全自洽，建议后续核对真题官方答案后修正基准数据。judge 分 0.82–0.85 说明答案实质质量尚可。

**3. 小模型的系统性短板画像**

31 题中仅 2 题因引用准确性硬伤失败、1 题（`2011_a22_01`，L0 judge 0.3）为边际波动。整体上 12B 本地模型在「分析框架组织」上已够用（judge 均值 0.72+），短板集中在**精确法条编号记忆**——这正是检索增强（L3 层级）和知识库能力可以针对性补强的方向，与路线图「证据化结论可定位权威来源」的核心度量一致。

**4. 自评偏差的量化提示**

`2008_a31_02` 案例显示：答案含法条编号幻觉时 judge 仍给 0.77+。阅读本报告所有 llm_judge 分数时应按「不校验引用正确性的流畅度/框架分」理解；citation_completeness（规则抽取）才是引用准确性的有效度量。后续若引入独立大模型做 judge（或人工抽检），预期 judge 均值会下移。

## 复现命令

```bash
# 生成+评判两阶段，按题缓存（$TMPDIR/mady_*_eval.json[.judge]），中断后重跑即断点续跑
MADY_LIVE_EVAL=1 MADY_EVAL_SUITE=p2a \
  MADY_EVAL_API_KEY=<omlx_key> MADY_EVAL_BASE_URL=http://127.0.0.1:8000/v1 \
  MADY_EVAL_MODEL=gemma-4-12B-it-8bit \
  go test ./agentcore/evaluate/benchmark/ \
  -run 'TestLiveDeepSeekEval$|TestLiveAgentBaselineEval$' -count=1 -timeout 6800s -v
```

全量冷跑约 60–80 分钟（本地单流吞吐限制）；缓存全命中时秒级完成（仅重打报告）。删除 `$TMPDIR/mady_deepseek_eval.json*` / `mady_agent_baseline_eval.json*` 即重新冷跑。

## 跑批 harness 工程说明（v0.8 新增）

为支撑全量跑批，`agentcore/evaluate/benchmark/` 的 live 测试做了以下增强（不含任何产品代码改动）：

- **双阶段结构**：生成与评判拆分为独立阶段，各自按题缓存（评判缓存 = 生成缓存路径 + `.judge`），任意阶段中断均可断点续跑
- **并发**：4 worker（`liveEvalWorkers`），先由主 goroutine 同步筛出 pending 集合再并发处理——修复了此前主循环直接读缓存与 worker 写缓存的 map 数据竞争（曾触发 `fatal error: concurrent map read and map write`）
- **超时**：单题 15 分钟（`liveCaseTimeout`），防止本地端点异常时整批挂死
- **长度上限**：`liveMaxResponseTokens = 8192`，消除超长输出的跑批 livelock（本地 ~55 字符/秒，>16K 字符的回答单窗口无法完成）

> 验证：`go vet` 双模块通过、`go test -race ./agentcore/evaluate/...` 全绿、`golangci-lint run` 双模块 0 issues。
