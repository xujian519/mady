原告/被告双视角 — 架构方案对比

三种实现双视角判定的架构模式，各有不同的适用场景

方案 A：统一图 + 视角参数（推荐）

单个 Pregel 图，以 Perspective 参数控制节点行为。原告和被告走同一拓扑，但每个节点的提示词和分析逻辑随视角切换。

load_input → claim_scope → feature_mapping → literal_infringement
                              ↘ equivalence → estoppel_check → defense_review → remedy → conclude

perspective=plaintiff → 各节点侧重"主张侵权成立的论证路径"
perspective=defendant → 各节点侧重"抗辩与规避的论证路径"

优势

  • 代码量最小，维护成本低
  • 拓扑一致保证判定连贯性
  • 与 inventiveness/enablement 模式完全对齐

劣势

  • 节点提示词需要同时维护两套逻辑
  • 部分节点（如 remedy）对原告/被告差异较大

方案 B：双图并行

两套独立的 Pregel 图，分别构建原告分析图和被告分析图，可独立调用或对比输出。

plaintiff_graph:
claim_scope → feature_mapping
→ literal_match → equivalence
→ infringement_conclusion
→ damage_calculation
→ remedy_strategy
defendant_graph:
claim_scope → feature_mapping
→ non_infringement_args
→ prior_art_defense
→ other_defenses
→ invalidity_strategy
→ risk_assessment

优势

  • 各自独立，提示词精准聚焦
  • 可并行执行，效率高

劣势

  • 代码量大（两份 graph + nodes）
  • 共享节点（如 claim_scope）需重复实现

方案 C：分层模型 — 核心判定层 + 视角策略层

底层为中立的核心侵权判定图，上层是两个视角策略 Agent，各自消费同一判定结果并从不同角度解读。

Layer 1 (核心判定): claim_scope → feature_mapping → literal → equivalence → estoppel → verdict
Layer 2a (原告策略): verdict + 原告参数 → infringement_strength / damage_model / injunction_feasibility
Layer 2b (被告策略): verdict + 被告参数 → defense_viability / invalidity_route / risk_exposure

优势

  • 核心判定中立客观，可复用
  • 视角层独立演化，互不干扰

劣势

  • 三层结构增加复杂度
  • 项目中无类似先例
