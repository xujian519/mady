核心引擎深度解析

agentcore — 模块内部架构与关键设计模式

LifecycleHook 拦截链

LifecycleHook 是 agentcore 最核心的扩展机制。Agent 的每次执行都沿以下路径触发钩子：

BeforeAgentRun → (Turn Loop: BeforeTurn → BeforeModelCall → [Model Call] → AfterModelCall → [if tool calls] BeforeToolExecution → [Tool Execution] → AfterToolExecution → AfterTurn ) → AfterAgentRun

消息持久化为独立钩子链，在每次 AddMessage 时触发：

Message Persist: BeforeMessagePersist → [Store] → AfterMessagePersist

钩子的实现者（LifecycleProvider）：

实现者所在模块拦截的钩子功能说明
GuardrailHook agentcore/lifecycle.go AfterModelCall 校验模型输出，拒绝不安全内容
ReasoningRouter agentcore/reasoning_router.go BeforeModelCall 按复杂度调节 thinking effort/budget
ReasoningStrategyRouter agentcore/reasoning_strategy.go BeforeModelCall 注入策略提示词 + 框架步骤
MemoryExtension memory/extension.go BeforeTurn, AfterTurn, BeforeAgentRun 三层记忆注入与持久化
compilerHook memory/compiler/extension.go BeforeTurn, AfterTurn 策略学习：ε-greedy 选策略 → 追踪执行 → 回馈
guardrail (领域) guardrails/levels.go AfterModelCall 三级护栏：拦截短语/风险关键词/审批关键词
CitationGate guardrails/citation_gate.go AfterModelCall S1 静态表 + S2 知识源双级核验
PlanModeExtension agentcore/planmode/extension.go BeforeToolExecution 计划模式工具门控
FileCheckpointExtension agentcore/filecheckpoint/extension.go BeforeAgentRun, AfterAgentRun 文件级快照与回退
Guardian guardrails/guardian/ AfterModelCall AI 安全审查熔断器
Extension 插件体系

Extension 接口是 Mady 的插件化基石。每个扩展可选择性地实现以下提供者接口：

接口方法注册到 Agent 的什么位置
Extension Name(), Init(ctx, agent), Dispose() 基础接口，所有扩展都必须实现
ToolProvider Tools() []*Tool agent.RegisterTools() — 注册工具
HookProvider BeforeHooks(), AfterHooks() agent.config.GlobalBefore/After — 添加到钩子链
MiddlewareProvider Middleware() []Middleware agent.config.Middleware — 中间件链
SystemPromptProvider SystemPromptSuffix() string 追加到 agent.config.SystemPrompt
TransformContextProvider TransformContext(ctx, msgs) []Message agent.config.TransformContext — 消息转换链
LifecycleProvider LifecycleHook() LifecycleHook agent.config.Lifecycle — 生命周期钩子链
EventSnapshotProvider SnapshotEvents() []Event 为新监听器提供快照事件
扩展注册流程：ext.Init(ctx, agent) → 类型断言检查 ToolProvider/HookProvider/LifecycleProvider 等 → 分别注册到 agent.RegisterTools()/agent.config.GlobalBefore/agent.config.Lifecycle。
销毁时倒序调用 ext.Dispose() — 先注册的后销毁。
Handoff 交接机制

Handoff 是 Mady 的多 Agent 协作核心。Router Agent 通过 transfer_to_* 工具将任务委派给领域 Agent。

组件文件说明
HandoffConfig agentcore/handoff.go 目标配置（Name/Description/Mode/AllowedSources/FallbackMsg/Invisible）
HandoffMode agentcore/handoff.go 两种模式：HandoffDelegate（子 Agent 运行后返回结果）或 HandoffTransfer（完全移交对话控制权）
HandoffContext agentcore/handoff_context.go 结构化交接上下文（意图/实体/最近消息），减少 token 消耗
HandoffResult agentcore/handoff_result.go 子 Agent 返回的结构化结果（Action/Result/Success）
SafeHandoff agentcore/handoff.go 基于 AllowedSources 白名单的运行时交接校验（isHandoffAllowed）
Invisible Handoff agentcore/handoff.go Invisible=true 时，子 Agent 事件不转发到事件总线，UI 无切换痕迹

Router Agent 委派流向（domains/router.go）：

Router (mady-router) → transfer_to_chat → Chat Agent (日常聊天) → transfer_to_assistant → Assistant Agent (任务执行) → transfer_to_patent → Patent Agent (专利分析) → transfer_to_legal → Legal Agent (法律分析)
推理策略路由

ReasoningRouter 按复杂度动态调节推理策略。三档复杂度（Low / Medium / High）对应不同的 thinking effort 和策略：

复杂度Thinking Effort默认策略框架步骤数典型触发场景
Low low StepByStep 3（理解→回答→验证） 简单问答、短文本（<200 字）
Medium medium StructuredAnalysis 4（分解→收集→分析→结论） 中等长度、无明显复杂关键词
High high VerifiedThinking 6（分解→研究→分析→综合→验证→结论） 包含分析/法律/专利/架构等关键词，或对话深度 ≥6 轮

6 种可选的推理策略（ReasoningStrategy）：

策略StrategyHint 核心指令
Default让模型自行选择推理方式
StepByStep按步骤逐步推理，每步标注【步骤N/总步数】
StructuredAnalysis结构化框架：分解→事实收集→法律分析→结论
Debate模拟多方辩论，综合各方观点
TreeOfThoughts思维树：关键决策点探索多个分支
VerifiedThinking每步自我验证：逻辑一致性、法条适用正确性
FirstPrinciples第一性原理：回归基础重建论证链
EventBus 事件系统
Broker[T] (agentcore/pubsub.go) — 类型安全的泛型事件总线。
支持两种发布语义：Publish（尽力投递，缓冲区满时丢弃）和 PublishMustDeliver（有界阻塞，默认 50ms 超时）。
每个订阅者独立缓冲通道（默认 4096），支持扇出到多个监听器。
Agent 执行循环
Agent.Run(input): → 加载配置 → BeforeAgentRun → (内循环: → BeforeTurn → ContextEngine 构建请求 (Chunked/Tiered/Truncate) → BeforeModelCall (→ ReasoningRouter/Strategy 注入) → Provider.Chat (LLM 调用) → AfterModelCall (→ GuardrailHook/Guardian/Guardrails/CitationGate 核验) → [有工具调用] Executor.ExecuteAll → BeforeToolExecution (→ PlanModeExtension 门控) → Middleware 链 (Timeout/Retry/Logging) → 执行每个 Tool.Func → AfterToolExecution → AfterTurn ) → AfterAgentRun → 返回 output

数据流与关键场景

核心业务流程的完整数据链路

1. Disclosure 交底书分析管线

disclosure/ 实现了 11 节点 Pregel 图，逐节点处理技术交底书。管线由 BuildDisclosureAnalysisGraphWithOpts 构建：

preprocess → extract_problem → merge_extractions → check_consistency ↕ extract_features → generate_keywords → retrieve_prior_art → check_novelty → generate_report → review_gate → draft_claims → __end__

实际上前三个提取节点并行执行：

preprocess → extract_problem → extract_features → extract_effects → merge_extractions

关键节点说明：

节点类型功能
preprocess纯函数文本清洗、分段、标准化预处理
extract_problemAgent 包装提取交底书要解决的技术问题
extract_featuresAgent 包装提取技术特征（功能/类别/重要性）
extract_effectsAgent 包装提取技术效果和有益效果
merge_extractions合并节点合并三个 Agent 输出，构建 PFE 三元组交叉引用
check_consistency条件节点一致性校验 → 不通过则注入 retry_feedback 回退到提取节点
generate_keywords纯函数从提取结果生成检索关键词
retrieve_prior_art检索器专利检索（支持可选的 DomainRetriever 注入）
check_noveltyAgent 包装基于现有技术评估新颖性/创造性
generate_reportAgent 包装生成结构化分析报告
review_gate中断门主动中断信号 → 等待人工复核（adopted/modified/rejected）
draft_claims纯函数根据审核结论撰写权利要求草案
条件回退机制：check_consistency 是条件边节点，通过 consistencyRouter 判断一致性校验结果： 通过 → 前进到 generate_keywords；不通过 → 将反馈写入 StateKeyRetryFeedback，回退到提取节点重试。
2. 引用核验门 (CitationGate)

双级引文核验机制，作为 LifecycleHook 在 AfterModelCall 阶段拦截：

级别核验方式核验对象
S1 静态表 citation_table.go · 82 条精校法条 检查模型引用是否在静态主题收录范围内
S2 知识源 citation_source.go · Composite 复合源 在多个知识源中验证引用的准确性
判定矩阵：S1 通过且 S2 通过 → 放行；S1 通过但 S2 不通过 → 标记警告；S1 不通过 → 返回明确错误信息。
3. 三级护栏系统
级别拦截功能适用场景
LevelLight 拦截 BlockedPhrases（恶意代码/攻击方法/非法入侵） 日常聊天
LevelStandard ↑ 以上 + 检测 RiskKeywords → 追加免责声明 专业咨询领域
LevelStrict ↑ 以上 + 检测 ApprovalKeywords → 挂起人工审批（SuppressPersist） 法律/专利等高风险领域

Guardian 熔断器（guardrails/guardian/）作为额外安全层，在护栏之上用独立 AI Agent 进行内容安全审查。

4. 三层记忆系统
层级实现功能
Layer 1: STM memory/extension.go 短期工作记忆 · 上下文窗口内的即时记忆
Layer 2: EWM memory/extractor.go · retriever.go 摘要工作记忆 · 提取历史关键信息供检索
Layer 3: LTM memory/sqlite_store.go · manager.go 长期记忆 · 持久化存储（SQLite 向量 + FTS5 全文）

策略学习编译器（memory/compiler/）：

Compiler 机制：在 BeforeTurn 时使用 ε-greedy 算法从策略池中选择最优策略，注入策略建议到系统提示词； AfterTurn 时追踪执行轨迹并回馈结果，更新策略的成功/失败计数。 支持策略衰减（DecayConfig）和探索率调参。执行轨迹采用环形缓冲区存储。
5. 用户请求完整处理链路
TUI / Server / A2A / MCP ← 用户输入 → Router Agent (domains/router.go) → 分类用户意图 (domains/classifier.go) → 调用 transfer_to_* 工具 → HandoffContext 构建 (提取意图/实体/最近消息) → SafeHandoff 白名单校验 (isHandoffAllowed) → 领域 Agent (Chat / Assistant / Patent / Legal) → LifecycleHook 链 → ReasoningRouter 分类复杂度 → 注入策略 → Provider (chatcompat / smartrouter) → Guardrails + CitationGate 核验 → 返回结果 → HandoffResult 包装返回 → TUI / Server 渲染输出
